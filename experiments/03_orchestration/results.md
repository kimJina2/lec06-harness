# 실험 03: Orchestration 3패턴 비교 실험

## 실험 설계

동일 작업(밈 번역 + 강의 분석 2단계)을 3가지 오케스트레이션 패턴으로 실행

| 패턴 | 구조 | 특징 |
|------|------|------|
| 싱글 | Agent → Task1 → Task2 | 단순, 순차, 컨텍스트 공유 |
| Planner+Executor | Planner → Plan → Executor(Task1, Task2) | 계획 단계 추가, 토큰 오버헤드 |
| 병렬 sub-agent | Orchestrator → [Agent1, Agent2] 동시 | 빠를 것이라 기대 — **결과는 반대** |

## 실행 환경

- 모델: `llama-3.1-8b-instant` (Groq)
- 실행 일시: 2026-05-13
- 작업 1: 밈 번역 (`"It's giving main character energy"`)
- 작업 2: Transformer 자막 ~400자 분석
- 1회 실행 결과

## 측정 항목

- 토큰 사용량 (total_tokens)
- 레이턴시 (초, wall clock)
- 실패 발생 레이어 (Planner / Executor / sub-agent)

## 결과

| 항목 | 싱글 | Planner+Executor | 병렬 sub-agent |
|------|------|-----------------|---------------|
| 레이턴시 (초) | **1.41** | 2.02 | 4.63 ⚠️ |
| 토큰 사용량 | **578** | 1133 (+96%) | 750 (+30%) |
| 실패 레이어 | - | - | - |
| 병렬 속도비 (vs 싱글) | 1.0x | 0.70x (느림) | **0.30x (3배 느림)** |
| 결과 퀄리티 (1~5, 주관) | 4 (truncated된 출력) | 4 (가장 완전한 JSON) | 3 (lecture 출력 가장 짧음) |

## 분석

### 핵심 발견 — **직관을 뒤집은 결과**

1. **"병렬 sub-agent가 가장 빠를 것"이라는 기본 가정이 무너졌다.**
   본 실험에서 병렬은 싱글보다 **3.3배 느렸다**. 원인 가설:
   - 작업 1개당 응답이 1초 미만으로 짧음 → ThreadPool 설정 오버헤드가 호출 시간보다 큼
   - Groq API의 동일 키 동시 호출 시 큐잉 (rate limit 안내가 wallclock에 반영)
   - 토큰 30% 증가 → 동일 응답을 약간 다르게 생성 (캐싱 효과 없음)

2. **Planner+Executor는 토큰 거의 2배인데, 결과 품질 향상은 미미.**
   Plan 단계 출력을 보면 "AI 번역 도구를 써서 번역" 같은 trivial한 단계 나열. 짧은 작업엔 Planner의 ROI 없음.

3. **싱글이 모든 면에서 우승.** 가장 빠르고, 토큰 가장 적고, 결과 품질도 동급. **2단계 짧은 작업에서 오케스트레이션은 오히려 손해.**

### 패턴별 적합한 작업

| 패턴 | 이 작업엔 이 패턴 |
|------|-----------------|
| 싱글 | **2~3단계의 짧고 결정적인 작업** (이번 실험이 정확히 여기) |
| Planner+Executor | 단계가 많고 분기가 있어 미리 계획이 가치 있을 때 (예: 5+단계 워크플로우) |
| 병렬 sub-agent | 각 작업이 충분히 무거워서 (단일 호출 5초+) 오버헤드를 정당화할 때 |

### "내 작업엔 이 패턴" — 회고

> 내가 lec06_harness에서 풀려는 병목(밈 번역, 강의 분석)은 1~2초짜리 짧은 작업이다.
> 이번 실험으로 확인: **싱글이 토큰 효율도 가장 좋고, 레이턴시도 가장 좋고, 결과 품질도 동급**.
>
> 병렬화는 "각 sub-agent의 호출이 5초+"로 무거워질 때 — 예를 들어 YouTube 자막 추출(네트워크 I/O)이나 긴 강의 자막(8000자) 분석 — 그때 다시 고려할 가치가 있다.
>
> Planner+Executor는 단계가 미리 정해진 이 종류 작업엔 과잉. 사용자가 처음 보는 작업을 던질 때(예: "X를 분석하고 Y로 정리해서 Z에 올려") 의미 있다.
>
> **교훈: 오케스트레이션 패턴 선택은 "작업의 크기와 결정성"에 좌우된다. 멋있어 보인다고 병렬·다단계를 기본값으로 쓰면 오히려 손해.**

### 노트북과의 연결

노트북의 `react_loop`는 **싱글 패턴**이었다. 이번 실험으로 그 선택이 옳았음을 확인. 만약 노트북에서 도구 호출 단계가 5개를 넘어갔다면 Planner+Executor를 고려할 가치가 있었을 것.

## 추가 실험 아이디어

- 작업 크기를 늘려서 (자막 8000자) 다시 비교 → 병렬이 언제부터 이기기 시작하는지 임계점 찾기
- 병렬 sub-agent에서 동일 키 대신 별도 API 키 사용 → rate limit 영향 분리
- Planner를 더 강력하게 만들어서 (5단계 작업 계획) 진짜로 가치 있는 케이스 만들기
